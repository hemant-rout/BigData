🚀 Part 1: Spark SQL Performance Tuning Tricks (Production-Level)

These apply directly to systems like Apache Spark running on Databricks.

🔹 1. Partition Pruning (Critical for Large Tables)

Your transactions table is partitioned by ingestion_date.

👉 Always filter on partition column:

WHERE ingestion_date >= current_date - INTERVAL 2 DAYS
Why it matters:
Avoids scanning TBs of unnecessary data
Reduces I/O drastically
🔹 2. Broadcast Join Optimization

Since customers_scd is small:

SELECT /*+ BROADCAST(c) */
...
FROM transactions t
JOIN customers_scd c
ON ...
When to use:
Dimension table < ~10GB (configurable)
Bonus:
SET spark.sql.autoBroadcastJoinThreshold = 50MB;
🔹 3. Optimize SCD Range Join

Range joins are expensive:

t.transaction_ts >= c.effective_from
AND t.transaction_ts < c.effective_to
Optimization techniques:
Pre-filter dimension:
WHERE is_current = true OR effective_to >= min_txn_date
Use join reordering awareness
Cache dimension table if reused
🔹 4. Z-Ordering (Databricks-Specific)
OPTIMIZE customers_scd
ZORDER BY (customer_id);
Why:
Co-locates related data
Speeds up joins & filters
🔹 5. Shuffle Partition Tuning

Default is often too high/low.

SET spark.sql.shuffle.partitions = 200;
Rule of thumb:
Too high → overhead
Too low → large partitions → OOM
🔹 6. Skew Handling (Very Important)

If customer_id is skewed:

Option 1: Salting
SELECT *, concat(customer_id, '_', rand(5)) AS salted_key
Option 2: Adaptive Query Execution (AQE)
SET spark.sql.adaptive.enabled = true;
SET spark.sql.adaptive.skewJoin.enabled = true;
🔹 7. Use Delta Cache / Caching
CACHE TABLE customers_scd;

or in PySpark:

df.cache()
Use when:
Table reused multiple times
🔹 8. Avoid Wide Transformations Early

Bad:

SELECT * FROM huge_table GROUP BY ...

Better:

Filter early
Select only required columns
🔹 9. File Size Optimization

Small files = performance killer

OPTIMIZE transactions;

Target:

128MB–1GB per file
🔹 10. Explain Plan Analysis (Must for Senior Engineers)
EXPLAIN FORMATTED
SELECT ...

Look for:

BroadcastHashJoin vs SortMergeJoin
Shuffle stages
Scan size

    .dropDuplicates(["event_id"])
